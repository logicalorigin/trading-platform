# Database Audit — Final Authoritative Report ("helium")

## 1. Executive summary

Helium is a single-user trading Postgres (~9.3/15 GB, hard 12-connection cap, three writers: Node API, Rust `market-data-worker`, Python `pyrus_compute`) whose binding constraint is pool/event-loop saturation by a handful of unbounded append firehoses, not table count. The v2 plan is directionally right (tier by lifecycle, bound Tier C, reclaim dead space, quarantine-rename over DROP, backup gate first) but contains **three high-severity factual errors that would either no-op the main retention fix or break a live cache**, plus several mis-tierings and omissions that leave genuinely-unbounded tables uncovered. Every file:line claim below was re-confirmed against source this pass. **Overall confidence: High on the static-code findings; Medium on disk/size/index-usage assertions that still need live Phase 0 `pg_stat` data (current sizes, `dead_pct`, `idx_scan=0`, and proof the Rust 6-hourly sweep + the in-process self-prunes are actually firing now).**

## 5. Where the plan is still WRONG or risky — and the fix

**A. [CRITICAL] `option_chain_snapshots` is NOT written by the Rust worker.** Plan 1b (45-49) and Phase 1.4 (193-195) say it is "still actively WRITTEN every ingest cycle" and instruct: "first remove the Rust persist call (`main.rs:201`/`ingest.rs:161`)." Reality: `persist_option_chain_snapshots` (`ingest.rs:161`) writes **`option_chain_latest`** via `upsert_option_chain_latest_tx` → `insert into option_chain_latest` (`ingest.rs:552`). There is **no `insert into option_chain_snapshots` anywhere.** The fn name and the `option_chain_snapshot` job kind are a naming trap. **Following the plan literally would delete the persist call that hydrates the live `option_chain_latest` cache (GEX reads it at `gex.rs:56`) — breaking trading data.** Fix: retire by unwiring `retention.rs:37` + `diagnostics.ts:4648` (prune) + `diagnostics.ts:2854` (monitor) + the cutover test, then quarantine-rename. It's a DELETE-only legacy drain, not a writer-retirement.

**B. [HIGH] Plan 1c mislabels `flow_events`, `flow_event_hydration_sessions`, `automation_diagnostics` as "Genuinely unbounded… NOT on any scheduled retention."** All three self-prune on the write path (§3). Fix: remove them from the gap list; the real task is verifying the in-process throttle fires when the write path idles.

**C. [HIGH] Phase 1.2's mechanism is broken.** It says close the gap for `execution_events`/`automation_diagnostics`/`shadow_position_marks`/`flow_events`/diagnostics "via a scheduled job invoking the existing TS `pruneDiagnosticStorage`." But `pruneDiagnosticStorage` filters the requested list to `PRUNABLE_CACHE_TABLES` (`diagnostics.ts:4667`, `requested.filter((t) => PRUNABLE_CACHE_TABLES[t])`). `execution_events`, `shadow_position_marks`, `signal_monitor_events`, `automation_diagnostics` are **not in the allow-list** → a scheduled call prunes none of them and **reports success.** Fix: split Phase 1.2 into (i) *schedule the existing helper* for the already-allow-listed tables, and (ii) *write NEW prune code* (add to the allow-list with correct cutoff columns, or add to `retention.rs`) for the unbounded ones.

**D. [MED] `balance_snapshots` mis-tiered Tier A "small, durable" and omitted from the gap.** It's an append firehose (§3). Fix: Tier C + retention on `as_of`. Do NOT merge with `shadow_balance_snapshots` (distinct real-vs-shadow source of truth).

**E. [MED] `position_lots` mis-tiered Tier A.** Write-dead duplicate of the position concept (real-broker analog of the dead `order_requests`/`broker_orders`/`execution_fills` family, `trading.ts:106-136`). Fix: drop the always-`[]` reader (`account.ts:5988`), fold into the dead-ledger retirement — no merge.

**F. [MED] `gex_snapshots` mis-tiered Tier B "(latest)".** `gex.rs:208` `ON CONFLICT (symbol, computed_at) DO UPDATE` → it accumulates a 30-day **history** (one row per compute cycle), not a one-row-per-symbol latest cache. Re-tier B→C so its mandatory `payload` jsonb reaches the Principle 4 strip pass.

**G. [MED] `market_data_ingest_jobs`, `shadow_balance_snapshots`, `signal_monitor_breadth_snapshots` are absent from every tier and the gap list** (§3). Add to Tier C with retention.

**H. [MED] Drift guard deferred too late.** Reconcile `historical_bars_provider_contract_quote_idx` now (Phase 1) and wire the generic drift check into `audit:guards` rather than scoping it to Phase 2 (§4-3).

**I. [LOW] `option_contracts` Tier-A "no retention needed" framing understates the never-deactivated expired backlog** (the same bloat the 20260611 composite was written to mitigate). Treat as a bounded-retention target (§3).
