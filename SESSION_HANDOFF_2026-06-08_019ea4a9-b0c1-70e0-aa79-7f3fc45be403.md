## Current Status

- User is planning how to turn PYRUS into a hosted SaaS where users connect
  their own broker accounts for execution without entering broker credentials
  into PYRUS.
- User decided v1 should go straight to automation-enabled broker connections,
  not a read-only/manual-first staged model.
- Created `docs/plans/broker-execution-platform-architecture.md` with the
  architecture, process flows, implementation phases, public API direction,
  test strategy, risks, and open questions.
- Locked v1 broker connection contract:
  `automation_trading_connection`.
- Required execution scopes are `read_account`, `read_positions`,
  `read_orders`, `read_executions`, `trade_submit`, and `trade_manage`.
  `trade_preview` and `order_update_stream` are preferred. `market_data` is
  disabled unless a broker requires it for order validation.
- Read-only/manual-only states are documented only as insufficient capability
  states, not v1 product modes.
- Terminal orders remain supported from an automation-active account, with
  per-order live confirmation. Automated strategy orders require strategy caps
  and kill switches.
- User confirmed v1 instrument/order-shape scope: stocks plus single-leg
  options only. Multi-leg options spreads and combo orders are deferred from
  v1 automation.
- User clarified execution should be broker/account-native: whichever broker
  account the user connects determines available order types, TIFs, sessions,
  routes, trailing stops, brackets, OCO/OSO, and cancel/replace rules.
- Updated plan to remove global PYRUS order-menu limits. PYRUS now discovers
  and enforces per-account capability maps while keeping normalized
  `OrderIntent` for caps, idempotency, audit, and reconciliation.
- User provided the website/portal access-point plan. Updated the broker
  architecture plan to make the portal the front door and app identity source:
  Better Auth portal session -> connected broker gate -> one-time platform
  handoff code -> handoff-derived platform session.
- User confirmed website dashboard shows read-only shadow/demo before broker
  connect, but `Launch platform` is gated behind `Connect Broker`; unconnected
  users cannot enter the terminal/platform.
- Interview confirmed the intended boundary: website handles PYRUS user login,
  first-time broker setup/activation, dashboard, and `Launch platform`; the
  launched PYRUS app handles the user workspace, broker-session continuity,
  broker reauth, manual trading, automation controls, and execution state.
- First-time broker connect belongs in the website portal dashboard/settings
  gate, not this platform. Routine broker reauth belongs in this platform via
  broker-hosted/OAuth flows. A return to the website portal is only for full
  reconnect/setup reset states, not ordinary reauth.
- Platform/broker adapters own durable broker authorization state, token/
  reference storage, reauth, and execution runtime. Portal owns
  customer-facing first-time broker-connect UX and launch handoff.
- Expanded Phase 0 with a detailed launch/auth contract:
  launch eligibility states, one-time handoff code API, handoff storage shape,
  server-to-server exchange, platform session properties, broker reauth
  ownership rules, audit event vocabulary, and redaction rules.
- Added `docs/decisions/ADR-001-portal-launch-handoff.md` to lock the
  portal-gated one-time handoff decision.
- Adversarial pass tightened the plan: strict referrer policy, strip handoff
  code from browser-visible URLs after exchange, portal back-channel exchange
  endpoint, revocation propagation from portal logout/suspension/broker
  disconnect, and no persistent raw broker token material in the portal.
- Added a supersession note: customer-facing v1 should not capture broker API
  keys, secrets, usernames, or passwords in PYRUS; broker authorization happens
  at the broker/OAuth/aggregator portal. IBKR Gateway remains a special
  connector path, not the general SaaS pattern.
