# GEX System Audit — 2026-06-16

Read-only audit of the GEX (dealer gamma exposure) system: backend math (`gex.ts`,
`gex-projection.ts`), data pipeline (`gex-universe-refresh.ts`, routes, providers, DB),
frontend models/hooks (`features/gex/*`), the GEX screen (`GexScreen.jsx`), and the
chart projection-cone overlay (`ResearchChartSurface.tsx`). No files changed.

## Verdict

The **core dollar-GEX formula is correct** (sign convention, ×100 multiplier, ×spot², ×0.01
per-1% — verified identical in all three implementations). The biggest risks are **not** the
core math — they're (a) the **projection cone's band/center semantics** misrepresenting the
projected range, (b) a **data-freshness pipeline** that can serve stale data as fresh and has
**no universe refresh scheduler**, and (c) **zero test coverage** on trading-decision math.

## Your two reported symptoms — answered

- **"GEX overlay lags the chart" — CONFIRMED, root cause nailed.** The cone is a React-state
  SVG overlay, not a native chart primitive. Path: `handleVisibleRangeChange` (`ResearchChartSurface.tsx:8755`)
  → `publishVisibleLogicalRange` → `setOverlayRevision` (`:8128`) → `useLayoutEffect` (`:9919`)
  → `buildGexProjectionConeSvgOverlay` → `setGexProjectionConeSvgOverlay` (`:9968`) → absolutely-positioned
  `<svg>` (`:12008`). It trails the canvas by ≥1 React commit per frame. No `attachPrimitive`/
  `ISeriesPrimitive` anywhere. **Fix:** reimplement as an `ISeriesPrimitive` + `IPrimitivePaneView`
  that draws on the chart canvas at paint time (coords resolved in the render loop). Substantial but correct.
- **"2 GEX cones" — NOT a bug / not a duplicate.** It's the intended **±1σ (inner) + ±2σ (outer)
  nested bands** plus a center line, rendered once as two `<path>` fills (`:12044-12055`). UX nit:
  they're under-differentiated (opacity + 0.15px stroke, no legend), so the nesting reads as a doubled cone.

## HIGH

**H1 — Projection cone bands are RND quantiles but the `center` is a GEX-pinned point, not the
distribution location.** `lower2/lower1/center/upper1/upper2` are fixed-probability quantiles
(15.87/84.13, 2.28/97.72%) of a Breeden–Litzenberger risk-neutral density, but `center` =
`adjustedCenter`, shifted toward peak-gamma/zero-gamma/walls and clamped into `[lower1, upper1]`
(`gex-projection.ts:648-671, 753-758, 888`). So the bands are **not symmetric or σ-consistent
with the center** — a trader reading "68%/95% around the projected center" misjudges skew. Fix:
either label the center as "GEX-pinned equilibrium, not distribution mean," or report band edges
as distribution quantiles independent of the center. **Confirm product intent.**

**H2 — ±2σ band silently clamps to the strike grid.** `quantile` returns a strike value clamped
to the available (interior) strikes (`gex-projection.ts:448-459, 408`); the chart path keeps only
±8 strikes around money (`gex.ts:2038`), so the 2.28/97.72% tails routinely fall outside and
`lower2/upper2` saturate to the edge strikes — the 2σ band becomes an artifact of strike selection,
not volatility. Fix: mark `quality:partial` when the quantile isn't interior to the grid, or widen
strike coverage for the band computation.

**H3 — Zero-gamma scanner: first-crossing semantics + inconsistent seeding, duplicated client-side.**
Both backend impls (`gex.ts:1062-1083`, `gex-projection.ts:617-636`) and the client twin
(`gexModel.js:279-302`) return the **first low-side crossing** (not nearest-spot), silently drop
multiple crossings, and the dashboard seeds the cumulative with `netGex[0]` while the projection
seeds `0`. The client recompute is redundant (the backend `/zero-gamma` field wins via
`useGexZeroGamma.js:180-187`). Fix: unify on `previousCum = 0`, document/choose the crossing
selection (nearest-spot?), and ideally delete the client recompute.

**H4 — GEX universe refresh has NO scheduler.** `refreshGexUniverseSnapshots` is invoked **only**
from the diagnostics POST route (`routes/diagnostics.ts:298`); there is no cron/`setInterval` in
`index.ts` (which does start the other workers). In DB-first mode, snapshots are only warmed
on-demand per requested symbol — the "500-symbol universe" is operator-driven only, so un-requested
symbols are perpetually stale/missing. Confirm intent; if pre-warm is required, wire a market-hours
worker like the others.

**H5 — `source.provider` is hardcoded `"ibkr"` even when spot/chain came from Massive.**
`gex.ts:1453` (and `:1542, :2103, :1663`) — spot can come from a Massive quote/reference close
(accepted at `:876`), but provenance always reports `ibkr`. The schema/type allow `massive` but it's
never emitted. Provenance is surfaced but **wrong**. Fix: derive provider from the actual source used.

**H6 — Stale snapshots served as fresh.** DB-first returns the persisted payload's **compute-time**
`isStale` (a 15-min rule baked in at compute) without re-stamping at serve time, behind a 60s
freshness gate (`gex.ts:1589-1602`; `market-data-ingest.ts:654-659`). The 60s gate and the 15-min
payload flag disagree, so a 14-min-old payload returned within the 60s window shows `isStale=false`.
No market-hours awareness. Fix: compute `isStale` at read time against an explicit threshold; reconcile
the two clocks.

**H7 — Zero test coverage for any GEX math.** No tests reference `contractGex`, `findGexZeroGamma`,
`buildGexProjection`, `buildGexLevels`, or the band quantiles. This is trading-decision code. Fix:
add unit tests for the sign convention (call vs put), a known-RND band check, multi-crossing
zero-gamma, all-positive/negative profiles, and tail-clamp detection.

**H8 — Heatmap legend & glossary colors don't match what renders.** Positive cells render **blue**
(`GEX_BULLISH_TONE → directionBuy → #168BFF`, `GexScreen.jsx:792`), but the legend swatches +
glossary say "green = call-heavy" (`HeatmapColorLegend.jsx:54-69`, `gexGlossary.js:154-159`), and the
legend's "Magnitude" strip is a red→neutral→green **gradient** while the heatmap actually encodes
sign-by-hue + magnitude-by-opacity (never a red-green spectrum). Fix: make legend/glossary match the
real encoding (blue/red, two single-hue opacity ramps).

## MEDIUM

- **Dashboard `contractGex` lacks a finite-number guard** the projection version has → a non-finite
  gamma can poison `netGex`/zero-gamma. `gex.ts:1025-1036`.
- **`r=0` fallback + hard-coded `q=0`** mis-center the RND forward for longer horizons (quality is
  downgraded but numeric bands are still emitted/consumed). `gex-projection.ts:260-262`, `gex.ts:1714-1724`.
- **Live path includes expired contracts** in `netGex`/zero-gamma; only the persisted path prunes
  (`gexRowExpirationTimeMs` used only in compaction/chart). Live vs DB-first inconsistency. `gex.ts loadGexDashboardData`.
- **`flowContext` is always `null`/`"unavailable"` on every path** — the flow-classification schema
  surface is unwired/dead. `gex.ts:1539-1540`. Confirm: finish or deprecate.
- **Live mode serves structurally-incomplete chains as `200 partial`** (e.g. 1 of 20 expirations);
  the Rust worker hard-fails on truncation but live doesn't. Define a min-coverage floor. `gex.ts`.
- **Market-closed = false "partial"; illiquid = misleadingly confident GEX.** Closed-market settlement
  spot trips `partial` solely due to `!quote`; zero-OI contracts are kept with no liquidity caveat.
  `gex.ts:869-882`.
- **`aggregateMetrics.ratio` returns `Infinity`** when there's no put GEX (`gexModel.js:310`) — guarded
  at one call site (`GexScreen.jsx:2062`) but should return `null`.
- **DEX `deltaShare` unnormalized when `refDelta` missing** → saturates to 0/25. `gexModel.js:544-545`.
- **`IntradayCard` runs `buildIntradaySnapshots` unmemoized every render** against a fresh-array prop
  (re-runs on the polling interval + resize). `GexScreen.jsx:2267`.
- **`gexDashboardCache` is unbounded** (no LRU/eviction; entries never removed). `gex.ts:249-256`.
- **Cone ±1σ vs ±2σ under-differentiated** (opacity + 0.15px stroke, no legend). `ResearchChartSurface.tsx:12044-12055`.
- **Desktop expiration `<select>` has no accessible name.** `GexScreen.jsx:1771`.

## Confirmed SOUND

- **Core dollar-GEX formula** (sign call+/put−, ×OI, ×100, ×spot², ×0.01 per-1%) — correct and
  identical in `gex.ts:1025`, `gex-projection.ts:534`, and `gexModel.js:235`.
- Black–Scholes guards (years=0/vol≤0 → intrinsic), Breeden–Litzenberger second-difference +
  de-discounting, density normalization and divide-by-zero guards (projection path).
- **Date/expiration/timezone handling is DST-safe and correct** (`gexDate.js` — verified by execution;
  NY-calendar-date via Intl, UTC-midnight day delta, expirations anchored 20:00 UTC consistently).
- GexScreen loading/empty/error/stale cascade + `DataUnavailableState` messaging; NaN/formatter guards;
  all major transforms memoized (except `buildIntradaySnapshots`); no dead controls.
- Serve-cache single-flight concurrency; worker DB-layer race safety (`FOR UPDATE SKIP LOCKED`, leases,
  unique dedupe/`(symbol,computed_at)` indexes); request timeouts (10s dashboard, chart budgets).

## Recommended priority order
1. **H1/H2 (cone band semantics + ±2σ clamp)** — most likely to mislead trading; decide intended semantics.
2. **H6 (stale-as-fresh) + H5 (wrong provenance) + H4 (no scheduler)** — data-trust pipeline.
3. **H3 (zero-gamma) + H7 (tests)** — correctness + lock it in.
4. **H8 (legend colors)** — quick, high-confusion copy/encoding fix.
5. **The cone-lag refactor (your reported bug)** — ISeriesPrimitive rewrite; bigger, isolated to the chart.
