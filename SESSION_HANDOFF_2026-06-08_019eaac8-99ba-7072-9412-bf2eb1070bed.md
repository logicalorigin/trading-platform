## Root Cause / Reasoning

- Observed source path:
  - Helper posts `/api/ibkr/activation/:id/login-key`.
  - Browser long-polls `/api/ibkr/activation/:id/login-key/read`.
  - Browser encrypts credentials and posts `/api/ibkr/activation/:id/login-envelope`.
  - Helper claims `/api/ibkr/activation/:id/login-envelope/claim`.
- Observed source issue:
  - The initial submit path calls `deliverIbkrLoginCredentials()` immediately after queuing launch.
  - The resume submit path was gated by `bridgeActivationLoginHandoffReady`, which comes from `session.runtime.ibkr.activation.latestActivation.loginHandoffReady`.
  - The `/api/ibkr/activation/:id/status` poll does not include `latestActivation.loginHandoffReady`; it only includes status/progress/insight.
- Inferred:
  - If the browser reloads, the original background delivery loop dies, or runtime diagnostics lag/drop, the UI can have a valid active launch but keep the primary credential action blocked until a separate runtime field arrives.
  - Removing the key-ready prerequisite is safe because the delivery function already waits for the helper key and handles cancellation/timeouts.
