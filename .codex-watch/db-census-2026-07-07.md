## 5. Pool: what should DB_POOL_MAX be, and is exhaustion cause or symptom?

**Symptom.** Evidence:
- Config: `DB_POOL_MAX` defaults to 12 on helium (lib/db/src/index.ts:194-230). The in-code rationale (lines 196-206, verified 2026-07-05) is explicit: helium's `max_connections` is 112 and the role limit is unlimited — 12 is a deliberate self-imposed cap because **the binding constraint is single-threaded result parsing on the event loop**, and a bigger pool "only piles more result sets onto that one thread."
- Runtime confirms it: ELU ≈ 1.0 with p95 event-loop delay 2-2.6 s (recorder memory samples) while Postgres backends sit idle in `Client:ClientRead` (pg_stat_activity probe: 0 active of 17 connections). The DB is waiting on the app, not vice versa. CPU profile: pg row parsing ~13%, GC 17.3%.
- The 231,263 `api-db-pool-acquire-slow` events/day are queueing for a resource held by queries whose *turnaround* is event-loop-bound; slow-query durations include that queue time (circular inflation).

**Recommendation: keep DB_POOL_MAX = 12.** Raising it would increase concurrent result-set parsing on the saturated loop and worsen latency. The census demand cuts (S1, S2, S4, S5, S6) remove the reads/writes that hold connections; only re-evaluate the pool number after demand drops and if acquire-waits persist with ELU < 0.8. A modest raise (e.g. 16) is safe DB-side (112 max_connections) but pointless until the event loop has headroom. Statement timeout 15 s / connect timeout 30 s (lib/db/src/index.ts:221-229) are sane; keep.
