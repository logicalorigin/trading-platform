## Executive verdict

Confirmed client-side freeze contributors:

- 429 pressure sheds could leave STA REST consumers with no recovery path because Signals fallback queries and PlatformApp signal profile/events queries used `retry:false`. Fixed with capped retry, `Retry-After` aware delay, and previous-data retention.
- SSE terminal reconnect already rebuilt the EventSource and therefore re-requested bootstrap, but large bootstrap/delta frames were applied as urgent React state. Fixed by wrapping matrix stream state commits in `startTransition`.

Refuted / not changed:

- Sparkline seed was not a permanent-blank path: it already chunk-published successful seed chunks and retried failed chunks. I only moved its `Retry-After` parser to the shared query helper.
- No server-side change was required. If the API does not already send `Retry-After` on route-admission 429s, adding that header remains recommended, but the client now falls back to capped retry delay.

## Suspect verdicts

### 1. 429 shedding without client recovery - confirmed

Observed:

- Generated react-query hooks only pass caller-supplied query options through to `useQuery`; no endpoint-specific retry is generated by default. Evidence: `lib/api-client-react/src/generated/api.ts:9880-9889` and `:10350-10359`.
- Generated custom fetch intentionally does not internally retry admission-shed responses (`x-pyrus-admission-action: shed`). Evidence: `lib/api-client-react/src/custom-fetch.ts:405-429`.
- PlatformApp signal profile/events queries had `retry:false`; now fixed at `artifacts/pyrus/src/features/platform/PlatformApp.jsx:3182-3192` and `:3318-3330`.
- SignalsScreen fallback profile/state/events/breadth queries had `retry:false`; now fixed at `artifacts/pyrus/src/screens/SignalsScreen.jsx:3439-3478`.
- Sparkline seed already had chunk publish + retry: `artifacts/pyrus/src/features/platform/MarketDataSubscriptionProvider.jsx:251-256`, `:856-858`.

Applied fixes:

- Added shared `parseRetryAfterMs` and `retryDelayWithRetryAfter` helpers in `artifacts/pyrus/src/features/platform/queryDefaults.js:17-34`; wired both normal and bars query defaults to honor `error.retryAfterMs` before capped fallback delay at `:36-55`.
- Tagged local `platformJsonRequest` errors with `retryAfterMs` from the response header at `artifacts/pyrus/src/features/platform/platformJsonRequest.js:52-67`.
- Added `retry: retryUnlessTimeout(2)`, shared retry delay, and `placeholderData: previousData` to STA REST consumers in PlatformApp and SignalsScreen.

Other signals-screen queries sharing this pressure-sensitive surface:

- Platform managed: signal profile, signal events.
- Signals fallback/unmanaged: profile, state route, events poll, breadth-history.
- Sparkline seed: already retried and kept runtime/last-good visual bars; now shares the same `Retry-After` parser.
