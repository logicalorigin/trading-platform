# WO-ALGO-REVIEW Report

## Verdict

No. The Algo control panel partially follows the redesign language, but it does not yet adhere to DESIGN.md because the right rail still leads as a controls/configuration surface more than an operational-readiness / blockers primary read.

## Hierarchy assessment

The current right rail starts with a narrow Wire Trail status band, then immediately shows timeframe controls, then halt controls, then the full settings region; diagnostics/readiness detail is isolated in a fixed footer below the scroll body. In `AlgoRightRail.jsx`, the controls body renders `WireTrailStatusBand`, `AlgoTimeframeControlBand`, `HaltStrip`, and `AlgoSettingsRegion` in that order (`artifacts/pyrus/src/screens/algo/AlgoRightRail.jsx:312`, `artifacts/pyrus/src/screens/algo/AlgoRightRail.jsx:316`, `artifacts/pyrus/src/screens/algo/AlgoRightRail.jsx:325`, `artifacts/pyrus/src/screens/algo/AlgoRightRail.jsx:334`), while diagnostics are separate and fixed-height at the bottom (`artifacts/pyrus/src/screens/algo/AlgoRightRail.jsx:356`).

There is a readiness signal in the broader page header: scan blocked, market data, risk, exposure, and attention strips are computed and rendered in the main Algo page (`artifacts/pyrus/src/screens/algo/AlgoLivePage.jsx:927`, `artifacts/pyrus/src/screens/algo/AlgoLivePage.jsx:1010`, `artifacts/pyrus/src/screens/algo/AlgoLivePage.jsx:1399`). But the work order is about the control panel/right rail. Inside that rail, the primary read is not "ready / blocked / why"; it is a stack of operational controls and configuration sections. `HaltStrip` computes an overall halt state (`artifacts/pyrus/src/screens/algo/HaltStrip.jsx:103`) but the header only colors the deployment name and does not render the overall state label/pill (`artifacts/pyrus/src/screens/algo/HaltStrip.jsx:568`, `artifacts/pyrus/src/screens/algo/HaltStrip.jsx:590`). The full settings region then renders multiple collapsible setting sections and editable fields (`artifacts/pyrus/src/screens/algo/AlgoSettingsRegion.jsx:3202`, `artifacts/pyrus/src/screens/algo/AlgoSettingsRegion.jsx:3245`), so configuration dominates the panel after a small status preface.

## Prioritized findings

1. high · HIERARCHY · The right rail lacks a clear primary operational-readiness read. It has status pieces, but no top-level "ready / blocked / next action" summary before the controls wall. The rail composition puts Wire Trail, timeframe controls, halt controls, then settings (`artifacts/pyrus/src/screens/algo/AlgoRightRail.jsx:312`, `artifacts/pyrus/src/screens/algo/AlgoRightRail.jsx:316`, `artifacts/pyrus/src/screens/algo/AlgoRightRail.jsx:325`, `artifacts/pyrus/src/screens/algo/AlgoRightRail.jsx:334`), and diagnostics are relegated to a separate lower frame (`artifacts/pyrus/src/screens/algo/AlgoRightRail.jsx:356`). Concrete fix: add a first rail band that summarizes operational readiness, active blockers, stale/offline sources, and the single next action; move timeframe/settings controls under a tertiary "Configure" grouping or leave them collapsed unless dirty.

2. high · HIERARCHY / ACCESSIBILITY / REJECTION RULES · `HaltStrip` computes state but hides much of it behind color and tooltips instead of visible labels. `overallHaltState` returns labels/colors (`artifacts/pyrus/src/screens/algo/HaltStrip.jsx:103`), but the header renders only the deployment name, colored by state (`artifacts/pyrus/src/screens/algo/HaltStrip.jsx:568`, `artifacts/pyrus/src/screens/algo/HaltStrip.jsx:580`, `artifacts/pyrus/src/screens/algo/HaltStrip.jsx:590`). Group rollups color the group label but do not show the promised rollup chip (`artifacts/pyrus/src/screens/algo/HaltStrip.jsx:633`). Individual controls render icon, label, dirty dot, and switch, but no visible `Armed / Active / Off / Forced` state pill (`artifacts/pyrus/src/screens/algo/HaltStrip.jsx:361`, `artifacts/pyrus/src/screens/algo/HaltStrip.jsx:397`). Concrete fix: render an overall status pill in the header, a visible rollup chip per group, and a small state pill per control; keep color as reinforcement, not the only cue.

3. medium · STATE COVERAGE · Disabled/loading/error/partial-stale states for the control panel are mostly implicit. `controlsReady` and `disabled` depend on focused deployment, baseline readiness, and pending mutations (`artifacts/pyrus/src/screens/algo/AlgoRightRail.jsx:240`, `artifacts/pyrus/src/screens/algo/HaltStrip.jsx:553`, `artifacts/pyrus/src/screens/algo/AlgoSettingsRegion.jsx:3185`), but the rail does not render a stable product state explaining "loading baseline", "no deployment", "save pending", or "control data unavailable" near the affected controls. Concrete fix: add fixed-height inline state rows for baseline loading, unavailable/error, pending save, and stale/partial data, with a clear recovery action where available.

4. medium · ACCESSIBILITY · Several controls are below the practical 44px target and lack visible focus styling in inline styles. The HaltStrip switch is 25px by 14px (`artifacts/pyrus/src/screens/algo/HaltStrip.jsx:232`), the halt numeric input suppresses outline (`artifacts/pyrus/src/screens/algo/HaltStrip.jsx:145`), and settings inputs also set `outline: "none"` (`artifacts/pyrus/src/screens/algo/AlgoSettingsRegion.jsx:173`, `artifacts/pyrus/src/screens/algo/AlgoSettingsRegion.jsx:843`). Concrete fix: wrap small visual switches in at least 44px keyboard/pointer hit targets where rail density allows, and route all interactive controls through a shared visible `:focus-visible` style instead of removing outlines.

5. medium · SEMANTIC COLOR · Directional call/put color is correctly routed through `toneForDirectionalIntent` in the strike ladder (`artifacts/pyrus/src/screens/algo/AlgoSettingsRegion.jsx:1363`, `artifacts/pyrus/src/screens/algo/AlgoSettingsRegion.jsx:1420`), but operational and financial colors are still assigned through local ad hoc maps and raw `CSS_COLOR.*` values. Examples: `HaltStrip` defines local status colors (`artifacts/pyrus/src/screens/algo/HaltStrip.jsx:46`), `WireTrailStatusBand` maps operational status directly to green/cyan/amber (`artifacts/pyrus/src/screens/algo/AlgoRightRail.jsx:31`), and overview metrics assign P&L/risk colors directly (`artifacts/pyrus/src/screens/algo/AlgoLivePage.jsx:1015`, `artifacts/pyrus/src/screens/algo/AlgoLivePage.jsx:1037`). Concrete fix: route operational, financial, and risk tones through semantic tone helpers, keeping direct token usage for neutral surfaces only.

6. low · STATE COVERAGE · Empty states exist but are inconsistent in product specificity. `No progressive steps` and `No wire rungs` are explicit enough to avoid a bare "no items" state (`artifacts/pyrus/src/screens/algo/AlgoSettingsRegion.jsx:2459`, `artifacts/pyrus/src/screens/algo/AlgoSettingsRegion.jsx:2581`), but they do not explain whether this is expected, disabled, or configurable. Concrete fix: use compact product states such as "Progressive trail off" or "No wire rungs configured" and pair them with the adjacent toggle/config affordance.

## Plan vs implementation

- The redesign plan says the panel should unify settings into dense compact cells with all sections open and "inputs + toggles only, no sliders" (`ALGO_RIGHT_RAIL_REDESIGN.md:26`, `ALGO_RIGHT_RAIL_REDESIGN.md:82`). The shipped settings region is collapsible by section (`artifacts/pyrus/src/screens/algo/AlgoSettingsRegion.jsx:3221`) and renders only open bodies (`artifacts/pyrus/src/screens/algo/AlgoSettingsRegion.jsx:3244`). It also implements DTE and exit controls as draggable slider-like handles (`artifacts/pyrus/src/screens/algo/AlgoSettingsRegion.jsx:1065`, `artifacts/pyrus/src/screens/algo/AlgoSettingsRegion.jsx:1967`, `artifacts/pyrus/src/screens/algo/AlgoSettingsRegion.jsx:2006`, `artifacts/pyrus/src/screens/algo/AlgoSettingsRegion.jsx:2048`).

- The plan calls for an always-expanded halt board with an overall status pill, group rollup chips, and per-row `Armed / Active / Off / Forced` state pills (`ALGO_RIGHT_RAIL_REDESIGN.md:99`, `ALGO_RIGHT_RAIL_REDESIGN.md:102`, `ALGO_RIGHT_RAIL_REDESIGN.md:110`, `ALGO_RIGHT_RAIL_REDESIGN.md:114`). The implementation has the board, groups, icons, color accents, and switches, but not the visible pills/labels described above (`artifacts/pyrus/src/screens/algo/HaltStrip.jsx:568`, `artifacts/pyrus/src/screens/algo/HaltStrip.jsx:633`, `artifacts/pyrus/src/screens/algo/HaltStrip.jsx:397`).

- The plan describes the strike ladder as two radio columns with radiogroups and `role="radio"` cells (`ALGO_RIGHT_RAIL_REDESIGN.md:152`, `ALGO_RIGHT_RAIL_REDESIGN.md:157`). The implementation uses checkbox semantics and supports multiple selected slots per side (`artifacts/pyrus/src/screens/algo/AlgoSettingsRegion.jsx:1197`, `artifacts/pyrus/src/screens/algo/AlgoSettingsRegion.jsx:1286`, `artifacts/pyrus/src/screens/algo/AlgoSettingsRegion.jsx:1313`, `artifacts/pyrus/src/screens/algo/AlgoSettingsRegion.jsx:1541`). This may be a valid product change, but the implementation and plan no longer match; either update the plan/spec to multi-select checkboxes or enforce single radio selection.

- The plan says the exit track is a horizontal P&L axis with markers edited via click-to-inline numeric input (`ALGO_RIGHT_RAIL_REDESIGN.md:167`, `ALGO_RIGHT_RAIL_REDESIGN.md:175`). The implementation is a richer price-path chart with draggable hard-stop/trail handles plus click-to-edit chips for only some markers (`artifacts/pyrus/src/screens/algo/AlgoSettingsRegion.jsx:1600`, `artifacts/pyrus/src/screens/algo/AlgoSettingsRegion.jsx:1848`, `artifacts/pyrus/src/screens/algo/AlgoSettingsRegion.jsx:1964`, `artifacts/pyrus/src/screens/algo/AlgoSettingsRegion.jsx:2167`). It may be useful, but it is not the shipped shape described in the plan.

- The plan's visual-verification handoff already flagged the remaining "compact toggle pocket" / hierarchy issue as open (`SESSION_HANDOFF_LIVE_2026-06-27_algo-control-panel-design-verify.md:50`). The current composition confirms that open item: Halt controls and settings are present, but the rail still does not start with a single readiness/blockers answer.
